Why We Built IndexNode in Rust

We evaluated Go, TypeScript, and Python before committing to Rust. Here's the reasoning — and what we'd choose again.

Distributed Systems Labs

Engineering Team

The Requirements That Drove the Decision

IndexNode processes blockchain events at scale — ingesting RPC streams, hashing content, building Merkle trees, writing to PostgreSQL, and pinning to IPFS, all concurrently. The requirements were clear:

  • Predictable latency under sustained load — no GC pauses
  • Memory safety without runtime overhead — blockchain data processing is memory-intensive
  • Excellent async I/O — most work is network-bound (RPC, IPFS, database)
  • Type safety for cryptographic operations — wrong hash types should fail at compile time
  • A strong ecosystem for Ethereum interaction

Why Not Go?

Go was the most serious alternative. It has excellent concurrency primitives, a mature ecosystem, and fast compile times. For most web services, it's the right choice.

The two places Go fell short for us were memory safety and the Ethereum ecosystem. Go's garbage collector, while excellent, introduces non-deterministic pause times that are problematic for real-time event streaming. And while there are Go Ethereum clients, the Rust ecosystem's ethers-rs library is significantly more comprehensive for our needs — ABI encoding, type-safe contract bindings, and wallet operations.

Why Not TypeScript?

TypeScript is where the Ethereum developer ecosystem lives — ethers.js, viem, The Graph's AssemblyScript indexers. We seriously considered it for the speed of iteration.

The runtime performance story for a high-throughput indexer isn't good enough. Node.js is single-threaded by default; working around that with worker threads adds complexity. The memory consumption of Node.js processes at scale is substantially higher than Rust. And TypeScript's type system, while excellent, can't provide the same compile-time guarantees for low-level operations that Rust's ownership model does.

Rust's Actual Advantages

The ownership and borrowing model forced us to reason about data lifetimes explicitly. This upfront cost paid for itself immediately in the concurrency layer: Rust's async/await with Tokio gives us thousands of concurrent tasks without data races — the compiler rejects programs that would race at compile time.

For cryptographic code specifically, the type system is invaluable. A SHA-256 hash and a Merkle root are both [u8; 32] arrays, but they're semantically different. Rust's newtype pattern lets us encode that distinction in the type system so mixing them up is a compile error, not a silent bug.

The performance characteristics are exactly what we needed: zero-cost abstractions, no GC, and memory usage that's predictable and low. Under peak indexing load, our workers run at consistent sub-millisecond per-event latency without the spikes that GC-based runtimes produce.

The Stack

  • Axum — ergonomic async web framework built on Tokio and Tower. Excellent middleware composability for our rate-limiting and auth layers.
  • async-graphql — the best GraphQL server implementation we found in any language. Schema-first with full subscription support over WebSocket.
  • sqlx — compile-time verified SQL queries against a live database. Typos in query strings are compile errors.
  • ethers-rs — full Ethereum JSON-RPC client with ABI encoding, typed contract bindings, and wallet support.
  • PostgreSQL — battle-tested relational database with excellent JSON support for storing event data and LISTEN/NOTIFY for pushing events to subscribers.

What We'd Do Differently

Rust's compile times are genuinely painful during development. A clean build of the full workspace takes several minutes on developer hardware. We addressed this with incremental compilation, cargo-watch for hot reloading, and splitting the workspace into smaller crates with stable interfaces.

The learning curve for team members new to Rust is real — the borrow checker requires a mental model shift that takes time to internalize. That said, the productivity investment pays back quickly: once code compiles, it usually works correctly on the first run in a way that other languages simply don't match.

We'd make the same choice again.
