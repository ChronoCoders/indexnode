Building Real-Time DeFi Analytics with IndexNode

A step-by-step walkthrough of building a production DeFi analytics dashboard — live swap volume, liquidity depth, and fee accrual — backed by cryptographically verified on-chain data.

Distributed Systems Labs

Engineering Team

What We're Building

DeFi protocols emit a rich stream of on-chain events: swaps, liquidity additions and removals, fee claims. This tutorial shows how to index those events with IndexNode and serve them to a frontend dashboard with sub-second latency via WebSocket subscriptions.

The stack: IndexNode GraphQL API for historical queries, WebSocket subscription for live events, and a lightweight frontend that aggregates volume and fee data. All event data is Merkle-verified before display.

Step 1 — Create a Blockchain Indexing Job

Start by authenticating and creating an indexing job for the contract you want to track. Use the createBlockchainJob mutation: 

mutation {
  createBlockchainJob(input: {
    contractAddress: "0x88e6A0c2dDD26FEEb64F039a2c41296FcB3f5640",
    chain: "ethereum",
    events: ["Swap", "Mint", "Burn"],
    fromBlock: 21000000
  }) {
    id
    status
  }
}

The job will enter the queue and begin processing. Depending on how far back fromBlock is, backfill may take several minutes. You can poll the job(id) query to check status.

Step 2 — Query Historical Events

Once indexing completes, query the accumulated events to build your baseline metrics: 

query {
  blockchainEvents(
    contractAddress: "0x88e6A0c2dDD26FEEb64F039a2c41296FcB3f5640",
    limit: 100
  ) {
    id
    eventName
    blockNumber
    transactionHash
    eventData
    contentHash
  }
}

The eventData field contains the decoded event parameters as JSON — for a Uniswap V3 Swap, this includes amount0, amount1, sqrtPriceX96, and liquidity. Use these to compute cumulative volume and current price.

Step 3 — Subscribe to Live Events

IndexNode pushes new indexed events over a WebSocket subscription. Connect with any GraphQL WebSocket client: 

subscription {
  blockchainEvents(
    contractAddress: "0x88e6A0c2dDD26FEEb64F039a2c41296FcB3f5640"
  ) {
    id
    eventName
    blockNumber
    eventData
    contentHash
  }
}

Each incoming event arrives within one block of being mined. Your frontend can increment running totals without polling — the subscription sends a message only when there's new data.

Step 4 — Verify Events Client-Side

Before displaying any value, verify the event's contentHash against the on-chain commitment. A minimal verification call: 

query {
  blockchainEvents(contractAddress: "0x...", limit: 1) {
    contentHash
    transactionHash
    blockNumber
  }
}
# Then call TimestampRegistry.getCommitment(contentHash) on-chain.
# Returns the block it was committed in — verifiable on any Ethereum/Polygon RPC.

For high-throughput dashboards, batch verification client-side by checking the Merkle root directly against the on-chain TimestampRegistry contract rather than making per-event API calls.

Aggregating Volume and Fees

Uniswap V3 swap events encode amounts as signed integers (amount0 and amount1) where a negative value indicates outflow and positive indicates inflow. To compute 24-hour volume, filter events by block range and sum the absolute value of token amounts across all swaps in that window.

Fee accrual can be derived from the pool's fee tier (stored in the contract) multiplied by the swap volume. For pools with dynamic fees, the fee is encoded directly in each Swap event's eventData.

Production Considerations

  • Cache verified event batches in your own database — don't re-fetch what you've already validated
  • Handle WebSocket reconnections gracefully by re-querying the last known block on reconnect
  • Store the on-chain Merkle root alongside your cached data so you can re-verify offline
  • Monitor credit balance via the creditBalance query and top up before jobs fail mid-run
