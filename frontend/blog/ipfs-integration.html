IPFS Integration: Permanent Data Availability

Every event indexed by IndexNode is now automatically pinned to IPFS. Your data doesn't disappear if we do.

Distributed Systems Labs

Engineering Team

The Problem with Centralized Storage

Every blockchain indexer stores event data in a centralized database. If the service shuts down, migrates infrastructure, or experiences a data loss event, your indexed history disappears. Applications that depend on that history — analytics dashboards, audit trails, compliance records — break with no recourse.

This is a fundamental architectural flaw. A service that positions itself as infrastructure for trustless applications shouldn't itself be a single point of failure.

IPFS as a Persistence Layer

The InterPlanetary File System (IPFS) addresses this by content-addressing data. A CID (Content Identifier) is derived from the hash of the content itself — not a location. Any node holding the data can serve it, and the CID guarantees you received exactly what was stored.

Starting today, every batch of events processed by IndexNode is serialized, hashed, and pinned to IPFS via Pinata. The CID is stored alongside each event in the IndexNode database and returned in API responses as ipfsCid.

How It Works

When a job completes, the worker serializes the full event batch as a JSON array, pins it to IPFS, and records the returned CID. Each individual event in the GraphQL API response includes its batch's ipfsCid. You can retrieve the full batch from any IPFS gateway using that CID — independently of IndexNode. 

query {
  blockchainEvents(
    contractAddress: "0x88e6A0c2dDD26FEEb64F039a2c41296FcB3f5640"
  ) {
    id
    eventName
    blockNumber
    contentHash
    ipfsCid
  }
}

With a CID in hand, you can fetch the underlying data from any public IPFS gateway or your own node. The content hash of what you retrieve will match the contentHash field in the API response — and that hash is committed on-chain via the TimestampRegistry contract, giving you an end-to-end verifiable chain of custody.

Querying IPFS Content Metadata

You can also query IPFS content metadata directly via the API to check pin status and size before fetching: 

query {
  ipfsContent(cid: "bafybeig...") {
    cid
    contentHash
    sizeBytes
    pinned
    createdAt
  }
}

Availability

IPFS storage is enabled for all Growth and Enterprise accounts. When creating an indexing job from the dashboard, toggle "Enable IPFS storage for indexed events" to activate it for that job. Historical jobs can be re-processed with IPFS enabled via the API.

Starter accounts receive IPFS storage for the first 1,000 events to evaluate the feature before depositing INC.

What This Means for Your Application

  • Your indexed data survives IndexNode outages — retrieve it directly from IPFS
  • CIDs are permanent — once pinned, the data is accessible indefinitely
  • Content addressing guarantees integrity — the CID encodes the hash of the data
  • On-chain Merkle roots tie IPFS content to immutable blockchain state

This completes the trustless data chain: blockchain events → indexed and hashed → Merkle root committed on-chain → batch pinned to IPFS. Every link is independently verifiable.
